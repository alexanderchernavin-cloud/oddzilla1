# oddzilla-android

Native Android client for Oddzilla. Storefront only (FE betting side
— no admin surface). Distributed as a self-hosted APK with built-in
update prompts; never published to the Play Store.

## Stack

- **Kotlin 2.0.21** + **Jetpack Compose** + **Material 3**
- **Navigation Compose** for routing
- **Retrofit 2** + **OkHttp 4** + **kotlinx.serialization** for the REST client
- **DataStore Preferences** for the persistent cookie jar
- **Coil** for image loading
- **AGP 8.7.3**, compileSdk **35**, minSdk **26** (Android 8.0+)

## First-time setup

1. **Open in Android Studio** (Hedgehog or newer) — `apps/mobile-android/`.
   Studio bundles a JDK and Gradle and will sync the project on first
   open. The Gradle wrapper jar is generated by the sync (gitignored).

2. **Drop in `local.properties`** if Studio doesn't auto-create one.
   Copy `local.properties.example` and edit `sdk.dir`.

3. **For release builds**, generate a signing keystore once:

   ```powershell
   keytool -genkeypair -v `
     -keystore "$env:USERPROFILE\keys\oddzilla.jks" `
     -alias oddzilla `
     -keyalg RSA -keysize 4096 -validity 25000 `
     -dname "CN=Oddzilla, O=Oddzilla, C=NL"
   ```

   Then copy `keystore.properties.example` → `keystore.properties`
   (gitignored) and fill in the storeFile / passwords. Without this
   the release build silently falls back to the debug keystore — the
   release script refuses to ship in that state, so check-in is safe.

   **Keep the keystore safe and backed up.** Lose it and you can never
   ship an update users can install over the existing build (signature
   mismatch). The keystore lives outside the repo on purpose.

## Day-to-day

```powershell
# Debug build on a connected device / emulator (Android Studio's
# green "Run" button does the same).
.\gradlew :app:installDebug

# Type-check + assemble without installing.
.\gradlew :app:assembleDebug

# Production-shape build (requires keystore.properties).
.\gradlew :app:assembleRelease
```

The debug variant uses the `cc.oddzilla.app.debug` application id and
appends `-debug` to versionName so it can sit alongside the release
build on the same device.

## Releasing

The release script bumps no files itself — it expects you to commit a
new `version.properties` first.

```powershell
# 1. Bump version.properties (versionCode +1, versionName = next semver).
.\scripts\bump-version.ps1 -New 0.2.0
git add version.properties && git commit -m "android: bump to 0.2.0"

# 2. Build, hash, and ship to the server.
.\scripts\release.ps1 -ReleaseNotes "Bug fixes and login improvements"

# Optional: force every old build to update before they can keep
# using the app.
.\scripts\release.ps1 -ReleaseNotes "Security fix" -Mandatory

# Optional: retire all builds below versionCode 7 even when the user
# previously dismissed the prompt.
.\scripts\release.ps1 -ReleaseNotes "Critical fix" -MinSupportedVersionCode 7
```

What the script does:

1. Reads `version.properties`.
2. Verifies `keystore.properties` is present (else aborts).
3. Runs `:app:assembleRelease`, producing
   `app/build/outputs/apk/release/app-release.apk`.
4. Computes the APK's SHA-256.
5. SCPs the APK to `team@178.104.174.24:/tmp/`, then sudo-installs it
   to `/srv/oddzilla-apk/oddzilla-<versionName>.apk`.
6. Writes `version.json` with the new `versionCode`, `apkUrl`,
   `sha256`, `releaseNotes`, and the `mandatory` /
   `minSupportedVersionCode` flags, then sudo-installs it to
   `/srv/oddzilla-apk/version.json`.

Caddy serves both files at `https://oddzilla.cc/app/*` (see
`Caddyfile` and `docker-compose.yml`). version.json has
`Cache-Control: no-store` so update prompts propagate within the time
it takes the user's app to next foreground; APKs are immutable so they
can sit in any caches indefinitely.

The bash variant (`scripts/release.sh "release notes"`) is identical
in behaviour — env vars replace named flags (`MANDATORY=1`,
`MIN_SUPPORTED_CODE=N`, `DRY_RUN=1`).

## How the update flow works

`MainActivity` triggers `UpdateController.check()` in a `LaunchedEffect`
on first composition. The controller:

1. Fetches `https://oddzilla.cc/app/version.json` via OkHttp.
2. Compares `manifest.versionCode` against `BuildConfig.VERSION_CODE`.
3. If newer (or current is below `minSupportedVersionCode`), surfaces
   `UpdateState.Available` to `UpdateGate`, which renders a
   `Material3 AlertDialog`.
4. On "Update now":
   - Probes `PackageManager.canRequestPackageInstalls()`. If false,
     opens `Settings.ACTION_MANAGE_UNKNOWN_APP_SOURCES` so the user
     can grant Install-from-this-source. They re-tap Update once back.
   - Otherwise downloads via OkHttp into
     `externalCacheDir/updates/oddzilla-<versionName>.apk`, hashing as
     it streams.
   - Verifies SHA-256 against the manifest. Mismatch → delete file,
     surface `Failed`.
   - Hands off to `Intent.ACTION_VIEW` with
     `application/vnd.android.package-archive` and a `FileProvider`
     content URI. The OS shows the install confirmation; on success
     the new APK launches over the old one.
5. Mandatory updates remove the dismiss button — the user must update
   to keep using the app. `mandatory=true` in version.json triggers
   this; so does `BuildConfig.VERSION_CODE < minSupportedVersionCode`.

### Testing the update flow without going to production

1. Bump `versionCode` in `version.properties` (don't ship — keep a
   debug keystore). Build & install on a test device.
2. Edit `/srv/oddzilla-apk/version.json` on the box manually:
   ```json
   {
     "versionCode": 999,
     "versionName": "0.99.0-test",
     "apkUrl": "https://oddzilla.cc/app/oddzilla-<an-existing-build>.apk",
     "sha256": "<that-build-s-sha256>",
     "releaseNotes": "test",
     "mandatory": false,
     "minSupportedVersionCode": 0
   }
   ```
3. Foreground the app. The dialog appears. Tap Update — the APK
   downloads, the system installer fires, the new build replaces the
   old one (signed by the same keystore, otherwise install fails).

## Architecture notes

- **CSRF**: every request stamps `Origin: https://oddzilla.cc` +
  `Referer: https://oddzilla.cc/` via `OriginInterceptor`. The API's
  CSRF plugin (services/api/src/plugins/csrf.ts) accepts these because
  they match `CORS_ORIGINS`. No API change required.
- **Auth**: cookie-based. The API sets `accessToken` (15 min) +
  `refreshToken` (30 d) cookies with `Domain=.oddzilla.cc; HttpOnly;
  Secure; SameSite=Lax`. `PersistentCookieJar` stores them in DataStore
  so a process restart doesn't bounce the user. A `TokenAuthenticator`
  catches 401s and silently calls `/auth/refresh` once before retrying
  the original request.
- **WebSocket**: same cookie jar feeds the `wss://oddzilla.cc/ws`
  upgrade so authentication piggybacks on the REST session. The
  `LiveOddsClient` keeps a single connection refcounted across screens
  — entering the sports list subscribes to every visible match, the
  match detail screen adds + removes its own match on mount/dispose,
  and unsubscribes are sent only when the last screen interested in a
  match goes away. Reconnect is exponential-backoff capped at 15 s.
- **Money**: bigint-as-string everywhere. Mobile never does arithmetic
  on micro values until parsing to `BigDecimal` at display time. See
  [`common/Money.kt`](app/src/main/java/cc/oddzilla/app/common/Money.kt).
- **Bet slip**: cross-match selections (max 20), Single (one ticket
  per leg) vs Combo (one ticket, product of odds) toggle, USDC/OZ
  currency switch persisted in DataStore (`oddzilla_slip` prefs).
  Place flow attaches a fresh `idempotencyKey` UUID per ticket so
  retries are safe. The bottom sheet auto-dismisses on success and
  surfaces typed errors from the API on failure.
- **Cashout**: quote-then-accept pattern matching the web. Tapping
  Cash out on an active ticket polls
  `GET /tickets/:id/cashout/quote`; the accept dialog roundtrips
  `POST /tickets/:id/cashout` with `expectedOfferMicro` so a stale
  quote is rejected server-side.
- **Manual DI**: Application-class service locator (`OddzillaDeps`)
  exposed via `LocalDeps` CompositionLocal. ViewModels take the deps
  container in their `Factory`. Hilt is intentionally not pulled in
  for an app this size.
- **No Firebase / Crashlytics / Analytics** — keep the binary clean
  for a self-hosted distribution. Crash reporting is a follow-up
  decision.

## Hard limits

- **Don't add cleartext network exceptions** beyond what the default
  Android network security config already gives us. Production traffic
  is HTTPS-only.
- **Don't add the `INSTALL_PACKAGES` system permission** — only
  `REQUEST_INSTALL_PACKAGES`. The former requires a system signature.
- **Don't ship a build signed with the debug keystore.** The release
  script aborts when `keystore.properties` is missing precisely to
  prevent this; the resulting APK won't install over a release build
  anyway (signature mismatch), but it would orphan the user.
- **Don't bypass the SHA-256 verification in `UpdateController`.** A
  silent fallback to "install anyway" would let a compromised CDN ship
  a tampered APK to every device on next launch.
